# 펌웨어 보안 분석 요약

## 1. 연구 범위

임베디드 장비 및 시스템 소프트웨어를 대상으로 보안 분석을 수행해 왔다. 주요 분석 대상은 공유기·IoT 펌웨어의 웹 관리면, 설정 처리 경로, 메시 프로토콜, 시스템 서비스 IPC 구조 등이다. 관심 범위는 단일 취약점 재현에 머물지 않으며, 입력 처리 구조, 인증 경계, 명령 실행 경로, 설정 반영 로직, 파일 복원·업데이트 파이프라인 등 시스템 수준의 공격면 전반을 포함한다.

주요 분석 축은 다음과 같다.

- 웹 엔드포인트, CGI, Lua/RPC 기반 관리면 공격면 식별
- 사용자 입력, 외부 환경 입력, 설정 저장값이 위험한 sink로 연결되는 데이터 흐름 추적
- 메시 동기화, 키 관리, IPC 등 설계 수준 취약점 분석
- 정적 분석 결과의 대량 펌웨어 triage 파이프라인으로의 일반화 및 도구화

## 2. 분석 방법

분석 과정은 단순 문자열 검색이나 단일 함수 확인이 아니라, 실제 데이터 흐름과 시스템 동작 구조를 기준으로 진행하였다. 일반적으로 다음 절차를 따랐다.

- 펌웨어 추출 및 파일시스템 복구
- 웹 엔드포인트, CGI, Lua/RPC, 서비스 데몬 식별
- 입력 지점, 설정 저장 지점, 명령 실행 지점, 메모리 복사 지점 추적
- 인증 경계, 검증/필터링, 재시작 이후 반영 여부 등 exploitability 판단
- 동일 vendor 내부의 반복 패턴 및 설계 상 recurrence 정리

동시에, 수작업 분석 결과를 대량 분석 가능한 형태로 일반화하기 위해 추출 실패 케이스, opaque container, partial extraction 상황에서도 의미 있는 triage 결과를 남기는 분석 파이프라인 개선을 병행하였다.

## 3. 대표 분석 사례

### 3-1. TOTOLINK A3002RU V3

TOTOLINK A3002RU V3에서는 서로 다른 기능 영역에서 반복적으로 나타나는 command injection 계열 구현 패턴을 분석하였다. 핵심은 특정 단일 기능의 버그보다, 사용자 또는 외부 환경 입력이 내부 설정 및 로컬 버퍼를 거쳐 shell command 템플릿에 직접 삽입되는 설계 습관의 반복이었다.

대표적으로 확인한 유형은 다음과 같다.

- 파일 업로드 처리 경로의 명령 실행 구조
- WPS/설정 관련 입력값의 명령 실행 구조
- 외부 무선 환경 입력(SSID)이 내부 설정 경로를 거쳐 실행에 도달하는 구조

이 사례를 통해 개별 취약점 보고를 넘어, 동일 제품군 내부에서 반복되는 shell-command construction anti-pattern을 recurrence case로 정리하였다.

### 3-2. TP-Link Deco XE75

TP-Link Deco XE75 계열에서는 단순 파라미터 검증 미흡이 아니라, 메시 동기화 설계 자체에 가까운 취약점을 분석하였다. 공유된 메시 그룹 키, 메시 인증 구조, 동기화 경로를 바탕으로 인증 우회 이후 설정 배포, 메시 내부 측면 이동, 후속 코드 실행 가능성까지 이어지는 구조를 정리하였다.

이 사례의 핵심은 다음과 같다.

- 단일 CGI 버그가 아닌 프로토콜 및 키 관리 설계 문제
- 메시 네트워크 전체에 영향을 줄 수 있는 시스템 수준 취약점
- 단일 노드가 아닌 다수 노드 관계를 고려한 공격 표면 분석

즉, 웹 관리 기능 취약점 수준을 넘어 메시 프로토콜 전반의 신뢰 구조를 다룬 사례이다.

### 3-3. ipTIME AX3000M

ipTIME AX3000M에서는 비공개 진단 인터페이스가 특정 버전대에서 명령 실행 경로로 이어지는 구조를 분석하였다. 버전별 기능 위치 변화와 제거 흐름까지 비교하여, 단순 발견이 아니라 변종 관계와 패치 방향까지 포함한 정리를 시도하였다.

주요 분석 포인트는 다음과 같다.

- 숨겨진 진단 기능의 실제 호출 구조
- 인증/게이트 로직의 실질적 보호 수단 여부
- 기존 공개 이슈와의 유사성 및 변종 여부
- 후속 버전에서의 제거 및 축소 여부

버전 차이와 공개 CVE와의 관계까지 포함한 비교 분석 사례라는 점에서 의미가 있었다.

### 3-4. Cudy WR3000E

Cudy WR3000E에서는 QoS 설정값이 nftables 규칙 생성으로 이어지는 흐름을 분석하였다. 이 사례는 전형적인 shell injection보다, 네트워크 정책 생성 로직과 실제 규칙 반영 경로의 안전성 검토에 가까운 사례였다.

주요 포인트는 다음과 같다.

- 사용자 설정값이 방화벽 규칙 템플릿에 반영되는 구조
- 설정 경로와 실제 nft 적용 경로의 연결
- 입력 검증 부족이 네트워크 정책 변조로 이어질 수 있는지 여부

즉, 웹 관리면 취약점을 넘어 정책 생성 엔진 관점에서 보안성을 본 사례였다.

### 3-5. Android Pixel 6 gpsd IPC

라우터·IoT 펌웨어 외에도 시스템 서비스 수준의 IPC 설계를 분석하였다. Android Pixel 6의 GPS 관련 서비스가 사용하는 IPC 채널과 권한 경계를 검토하여, 접근 가능한 주체와 실제 서비스 영향 범위를 구분하였다.

이 사례의 핵심은 다음과 같다.

- 단순 메모리/명령 실행 취약점이 아닌 IPC 설계와 권한 모델 분석
- DAC와 SELinux를 함께 고려한 현실적 위험도 해석
- 취약해 보이는 구조와 실제 악용 가능성을 구분하는 보수적 평가

## 4. 축적한 분석 역량

### 4-1. 데이터 흐름 중심 분석

입력값이 어디서 들어와 어떤 중간 상태를 거쳐 실제 위험한 sink에 도달하는지 구조적으로 추적하는 분석 방식을 유지해 왔다. 단순 키워드 매칭이 아니라 다음과 같은 축을 지속적으로 구분하며 분석을 수행하였다.

- 직접 HTTP 입력 여부
- 설정 저장 후 재시작을 거치는 deferred path 여부
- 외부 환경 입력 여부
- 인증 이후에만 reachable한지 여부
- 필터링/escaping이 실제로 존재하는지 여부

### 4-2. 반복 패턴 및 설계 문제에 대한 관심

개별 취약점 1건의 재현보다, 동일 vendor 또는 제품군에서 반복되는 코드 습관과 설계 패턴을 묶어 해석하는 접근을 선호하였다. 대표 예시는 다음과 같다.

- shell command 템플릿에 문자열을 직접 삽입하는 반복 패턴
- 숨겨진 진단 경로의 재사용 및 변형
- 메시 키 관리와 동기화 설계 문제

### 4-3. 보수적 판정 기준

무조건 취약점으로 단정하기보다, 다음 요소를 기준으로 false positive를 줄이는 방향의 분석을 지향하였다.

- 입력 제어 가능성
- 인증 경계
- 실제 sink 도달 여부
- 검증 및 제한 로직 존재 여부

이 기준은 수작업 보고서 작성뿐 아니라 대량 펌웨어 triage 도구 개선 과정에도 동일하게 적용하였다.

### 4-4. 분석 결과의 도구화

개별 보고서 작성에 그치지 않고 펌웨어 분석 파이프라인 자체를 개선하여 다음 기능의 자동화·반자동화를 진행하였다.

- opaque/partial firmware의 구조화된 추출 상태 정리
- 웹면, 서비스, 후보 sink 자동 식별
- 입력-핸들러-sink의 partial chain 수집
- 취약점 확정 전 단계의 conservative triage

즉, 수작업 분석 경험을 다시 도구 성능 개선으로 환원하는 방향의 연구를 함께 진행해 왔다.

## 5. 연구 성향

지향하는 분석 성향은 다음과 같다.

- 단일 exploit 재현보다 구조적 원인 파악 우선
- 과장된 claim보다 보수적 근거 제시 우선
- 개별 버그보다 vendor 수준 패턴과 일반화 가능성 중시
- 정적 분석과 자동화 도구 개선 병행
- 분석 결과의 재사용 가능한 보고서, 평가셋, triage 큐 형태 축적

## 6. 확장 방향

향후에는 다음 방향으로 연구를 확장하고자 한다.

- 대규모 펌웨어 corpus 대상 자동 추출 및 triage 안정화
- LLM 수준의 보수적 판단을 반영한 분석 엔진 고도화
- partial extraction 상황에서도 유의미한 보안 판단을 내리는 체계 구축
- 웹 관리면, 설정 파서, 업데이트·복원 경로에 대한 일반화된 분석
- 설계 수준 취약점과 구현 수준 취약점을 함께 다루는 분석 프레임워크 구축

## 7. 요약

라우터·IoT·시스템 서비스 펌웨어를 대상으로, 웹 입력, 설정 경로, 메시 프로토콜, IPC, 파일 처리 흐름을 추적하여 개별 취약점과 반복 설계 패턴을 분석해 왔다. 동시에 이러한 분석 경험을 대량 펌웨어 분석 도구로 일반화하는 작업을 병행해 왔으며, 실전 취약점 분석과 자동화 도구 개선을 함께 수행하는 방향의 연구를 이어가고 있다.
