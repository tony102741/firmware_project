# Structure Review: Research + CVE Hunting Pipeline

This note is a point-in-time structure review from a portable-development pass.
It is not authoritative for the desktop WSL workspace, which may contain additional local run bundles, regenerated corpora, Ghidra state, and other non-Git artifacts.

## Current Shape

The project already has the right high-level loop:

1. `src/pipeline.py`
   - extracts firmware, creates run directories, writes `manifest.json`, `results.json`, logs, and dossiers.
2. `src/main.py`
   - produces candidates, CVE triage fields, manual-review hints, candidate dossiers, and structured output.
3. `src/batch/batch_regression.py`
   - runs corpus-wide regression, updates corpus metadata, emits batch summaries, and can trigger LLM/review follow-up.
4. `src/review/llm_review.py`
   - converts result bundles into evidence packets and gold-label stubs.
5. `src/review/llm_review_infer.py`
   - runs heuristic/OpenAI/hybrid packet classification.
6. `src/review/manual_review_queue.py`
   - prioritizes cases for human review using engine state, LLM predictions, and prior manual labels.
7. `src/review/llm_review_eval.py`
   - compares predictions against gold/manual labels.
8. `src/research_tools/ledger.py`
   - validates candidate-ledger entries.

This is enough for an iterative loop:

```text
corpus -> pipeline runs -> results/dossiers -> LLM packets -> predictions
       -> manual review queue -> human labels -> eval -> tool improvements
```

## What Looks Strong

- Corpus metadata is explicit and already tracks extraction status, analysis status, run id, success quality, probe readiness, and blob family.
- Batch regression has a usable path for corpus-scale experiments and can chain into LLM packet generation.
- LLM review is evidence-grounded rather than free-form. Packets, predictions, and gold stubs are JSONL.
- Manual review queue encodes triage pressure: opaque containers, weak candidates, missing links, and LLM disagreement.
- Existing research artifacts already contain cross-firmware recurrence, architecture-level reports, and candidate-ranking notes.

## Structural Gaps

### 1. Run Artifacts Are Not Reproducible On This Machine Yet

`research/corpus/firmware_corpus.jsonl` points to 54 completed runs, but local `runs/` is currently empty. That means corpus rows and old review packets exist, but the referenced `results.json` and `manifest.json` artifacts are not locally available.

Impact:

- `llm_review.py --corpus ...` cannot regenerate packets from current run artifacts unless runs are copied or rerun.
- It is hard to audit whether old packets match current analyzer code.
- Paper reproducibility is weaker because corpus metadata and run bundles are separated.

### 2. Gold Labels Are Not Actually Gold Yet

`research/review/llm/llm_review_gold.jsonl` is labeled as autogenerated stubs. Current heuristic evaluation shows 100% accuracy against those stubs, which mostly proves schema consistency, not model quality.

Impact:

- Accuracy numbers are not publishable as human-review performance.
- The LLM loop needs a clear split between `engine_stub`, `llm_prediction`, and `human_label`.

### 3. Corpus Identity Has At Least One Duplicate

Current corpus summary shows one duplicate `corpus_id`:

- `d-link-dir-x3260-a1-101b05`

Impact:

- Review packets, predictions, and batch summaries can collide by `review_id`.
- Cross-version recurrence statistics may silently merge distinct files.

### 4. Source Provenance Is Mostly Missing

Corpus rows are missing:

- `source_url`: 54/54
- `source_page`: 54/54
- `release_date`: 30/54
- `arch`: 50/54

Impact:

- Top-conference methodology and artifact evaluation will need stronger provenance.
- Vendor disclosure/CVE writeups need exact firmware source and version evidence.

### 5. CVE Hunting Is Not A First-Class State Machine Yet

The code has CVE triage, dossiers, and Ghidra target prep, but there is no single candidate lifecycle record for:

```text
surfaced -> reviewed -> reversed in Ghidra -> PoC attempted -> vendor reported -> CVE assigned/rejected
```

Impact:

- CVE work can happen, but it is easy for evidence to spread across reports, Ghidra notes, and manual ledgers.
- Later paper claims need a clean trail from automated candidate to confirmed/rejected outcome.

## Recommended Next Steps

### Priority 1: Make A Canonical Experiment Bundle

Create a stable output directory per experiment:

```text
research/experiments/<date-or-name>/
  corpus.snapshot.jsonl
  batch_regression_summary.json
  llm_review_packets.jsonl
  llm_review_predictions_heuristic.jsonl
  llm_review_predictions_hybrid.jsonl
  manual_review_labels.jsonl
  manual_review_queue.jsonl
  eval.json
  notes.md
```

Keep this directory local/manual-sync if it grows large.

### Priority 2: Separate Label Types

Use explicit files:

- `engine_gold_stubs.jsonl`: autogenerated labels from current engine state.
- `llm_predictions_*.jsonl`: model outputs.
- `human_labels.jsonl`: reviewed labels only.

Never report model accuracy against autogenerated stubs as human accuracy.

### Priority 3: Add A CVE Candidate Ledger Schema

Track each serious candidate with:

- stable candidate id
- corpus id and run id
- binary/path/function/handler
- source-to-sink evidence
- auth/reachability state
- Ghidra MCP notes path
- exploitability verdict
- disclosure status
- CVE/vendor status

### Priority 4: Normalize Corpus Metadata

Fix duplicate corpus ids and backfill source pages, release dates, and architecture. This directly improves paper methodology.

### Priority 5: Recreate Or Copy Run Bundles

Either copy the relevant `runs/` directories from the desktop or rerun a smaller paper-grade subset on this machine. Start with 10-15 samples covering:

- known rootfs success
- blob/container success
- fallback success
- at least three vendors
- at least one strong CVE-hunting candidate

## Immediate Working Recommendation

Do not run the full corpus first. Start by making a clean, small, reproducible experiment slice and force the whole loop through it:

```bash
python3 src/batch/batch_regression.py research/corpus/firmware_corpus.jsonl \
  --limit 10 \
  --workspace-root /tmp/fw_batch_regression \
  --json-output research/experiments/smoke_20260513/batch_regression_summary.json \
  --with-llm \
  --llm-provider heuristic \
  --with-review-queue
```

Then manually inspect the top review queue rows and promote only strong cases into a CVE candidate ledger.
